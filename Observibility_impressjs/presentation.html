Observability is all about figuring out what is going on in a system

What Is Observability

Observibility and monitoring are different. Monitoring is a list of failure conditions that are known.

Observability involves emitting any and all data points that you may need in the future.

In a very real way Observability extends monitoring even though they are distinct items

What Makes A System Observable?

Observibility enables you to answer previously unknown questions

Observibility aims to provide highly granular insights into the behavior of systems along with rich context

A system built with Observability will be ruled out the quickest in the event of a failure

Most of your time will be spent working on systems without the same level of data

Get A Baseline!

The only way to know how you are doing is to know where you came from

Engineers must consciously code their application to emit metrics

Metrics gives you a time where something _may_ have gone wrong

It does not show you _what_ went wrong

The problem with monitoring, you are only looking for known problems

Monitoring: KISS

Some Tracing Required!

The goal is to emit enough data to allow for easy debugging

You don't know what you will need to handle unforseen problems

Tracing is the end-to-end journey of a user request

Tracing is a series of spans

A span is a representation of each individual component in the request chain

Applications need to be using a tracing library for maximum effectiveness

The Truth Is In The Logs

You need to have a solid logging solution

Nobody reads logs for fun

Logs are the only thing that can tell you what went wrong
